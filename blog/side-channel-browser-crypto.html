Attaques Side-Channel sur la Crypto Navigateur

Timing attacks, cache attacks et pourquoi WebCrypto API n'est pas magique. Les pièges de la cryptographie côté client.

Qu'est-ce qu'une attaque side-channel ?

Une attaque par canaux auxiliaires n'attaque pas l'algorithme cryptographique lui-même, mais son implémentation. Elle exploite des fuites d'information via le temps d'exécution, la consommation électrique, ou les accès mémoire.

Dans un navigateur : JavaScript peut mesurer le temps avec performance.now(). Même avec une résolution réduite (pour contrer Spectre), des attaques statistiques restent possibles.

Timing attacks

Comparaison de strings vulnérable

// VULNÉRABLE : comparaison caractère par caractère
function checkPassword(input, stored) {
    if (input.length !== stored.length) return false;
    for (let i = 0; i < input.length; i++) {
        if (input[i] !== stored[i]) return false;  // Fuite!
    }
    return true;
}

// L'attaquant peut deviner caractère par caractère
// en mesurant le temps de réponse

Comparaison constante

// SÉCURISÉ : temps constant
function safeCompare(a, b) {
    if (a.length !== b.length) return false;
    let result = 0;
    for (let i = 0; i < a.length; i++) {
        result |= a.charCodeAt(i) ^ b.charCodeAt(i);
    }
    return result === 0;
}

HMAC timing attack

Même avec une comparaison sécurisée, le calcul du HMAC lui-même peut fuiter :

// Mesure du temps de vérification HMAC
async function timingAttack(message, partialMac) {
    const times = [];
    
    for (let byte = 0; byte < 256; byte++) {
        const testMac = partialMac + String.fromCharCode(byte);
        const start = performance.now();
        
        await verifyHMAC(message, testMac);
        
        const elapsed = performance.now() - start;
        times.push({ byte, elapsed });
    }
    
    // Le byte correct prend plus de temps (plus de comparaisons)
    return times.sort((a, b) => b.elapsed - a.elapsed)[0].byte;
}

WebCrypto API

WebCrypto délègue les opérations au navigateur (code natif), ce qui réduit MAIS N'ÉLIMINE PAS les risques :

  • Les implémentations natives peuvent avoir leurs propres fuites
  • Le code JS autour de WebCrypto peut introduire des vulnérabilités
  • La gestion des clés en JS expose la mémoire
// WebCrypto : mieux mais pas parfait
const key = await crypto.subtle.importKey(
    'raw',
    keyBuffer,
    { name: 'AES-GCM' },
    false,  // non-extractable
    ['encrypt', 'decrypt']
);

// La clé est en mémoire JavaScript avant l'import
// Un attaquant avec accès mémoire peut la récupérer

Mitigations

Côté code

  • Utiliser des implémentations constant-time
  • Privilégier WebCrypto aux libraries JS pures
  • Minimiser le temps de vie des clés en mémoire

Côté navigateur

  • Réduire la précision de performance.now()
  • Isoler les contextes (Site Isolation)
  • Désactiver SharedArrayBuffer si non nécessaire

Conclusion

La cryptographie côté client est intrinsèquement risquée. Même avec WebCrypto, le contexte JavaScript est hostile. Pour les opérations sensibles, préférez les opérations côté serveur ou des environnements plus contrôlés.
